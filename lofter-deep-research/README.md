# LOFTER Deep Research

> 面向算法和数据同学的算法预研、业务分析、实验回收工作台

## 📖 项目简介

本项目是 LOFTER 团队的 **深度研究（Deep Research）** 工程仓库，核心目标是为算法和数据同学提供一个结构化的工作空间，用于：

1. **业务分析** — 对 LOFTER 业务指标、用户行为、内容生态等进行数据探索与洞察
2. **算法预研** — 在正式立项前，快速验证算法思路、跑通 PoC、评估可行性
3. **实验结果回收** — 系统化地记录 A/B 实验、离线评估等结果，沉淀可复用的结论

## 🚀 两种协作模式

本项目支持两种 AI 辅助的开发/研究模式，请根据场景灵活选择：

### 模式一：Vibe Coding（探索式对话编程）

适用于 **开放性探索、头脑风暴、快速原型** 等场景。

**工作流程：**
0. 先写个初版 proposal.md，输出一份初步的需求文档，大致包括：背景、目标、约束条件、验收标准
1. 与 AI 进行多轮自由对话，逐步明确分析思路或算法方案
2. 在对话过程中迭代代码、调整方向
3. **关键步骤 — 总结存档**：每次 Vibe Coding 会话结束后，要求 AI 对本次多轮讨论进行总结，输出一份结构化的会话摘要，包括：
   - 🎯 讨论目标与背景
   - 💡 核心结论与发现
   - 🔀 关键决策与转折点
   - 📝 后续 TODO
   - 🔗 相关代码/数据产出物
4. 将总结文档存档至 `notebooks/` 目录，命名格式：`<主题>`

```
alg/
├── <项目名>/
│   ├── proposal.md           # 需求文档（核心）
│   ├── technical_design.md   # 技术方案（核心）
│   ├── archive.md            # 存档文档（核心）
│   ├── *.py                  # 代码实现
│   ├── requirements.txt      # Python 依赖
│   └── ...
└── ...
```


### 模式二：Spec Coding（规范化需求驱动开发）

适用于 **目标明确、需要严谨设计** 的分析或算法任务。

**建议按照 [OpenSpec](https://km.netease.com/v4/detail/blog/261209) 流程开发：**

1. **需求澄清（Clarify）** — 明确分析/预研的目标、输入输出、约束条件
2. **方案设计（Design）** — 输出技术方案文档，包括数据流、算法架构、评估方案等
3. **任务拆解（Plan）** — 将方案拆解为可执行的子任务清单
4. **逐步实现（Implement）** — 按计划逐步编码实现
5. **验证回顾（Review）** — 验证结果，回顾过程，总结经验

**存档要求：**

OpenSpec会自动生成文档。



## 📂 项目结构

```
lofter-deep-research/
├── README.md              # 项目说明（本文件）
├── notebooks/             # Jupyter Notebooks

│   └── engineer/          # 工程实验（数据管道、工具验证、性能测试等）
├── scripts/               # 工具脚本（数据处理、批量任务、自动化等）
```

## 📋 使用指南

### 快速开始

1. 克隆仓库：
   ```bash
   git clone <repo-url>
   cd lofter-deep-research
   ```

2. 选择协作模式，开始工作：
   - **Vibe Coding**：直接与 AI 对话，探索问题，结束后记得存档总结
   - **Spec Coding**：先撰写 Spec，再按计划实现，全程文档化

### 存档复用

所有存档文档的核心价值在于 **可复用**：

- 🔍 **查阅历史**：遇到类似问题时，先搜索 `docs/` 目录，看是否有前人的分析和结论
- 🔄 **避免重复劳动**：站在之前研究的基础上继续深入，而非从零开始
- 📊 **沉淀方法论**：逐步积累团队的分析范式和算法经验库

## 🤝 贡献规范

1. 每次研究/分析 **必须** 产出存档文档
2. 代码提交前确保可运行，关键步骤有注释
3. 数据文件不要直接提交到 Git，使用 `.gitignore` 排除或注明数据获取方式
4. 命名清晰，让其他同学能通过文件名快速了解内容
