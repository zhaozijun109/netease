# 新回用户行为策略实现说明

## 概述

基于现有的 `ads_push_interest_tag_content_browse_di` 和 `ads_push_interest_tag_content_no_browse_di` 策略，实现了两个新的回用户行为策略，旨在提升用户召回效率和覆盖率。

## 策略一：回流用户历史兴趣内容推荐

### 基本信息
- **文件位置**: `analysis/jobs/push-1200/`
- **调度时间**: 12:00
- **MessageType**: 30
- **Priority**: 2
- **ExecutionId**: `return_user_history_interest`
- **Policy**: `return_user_history_interest`

### 策略描述
包含回流用户次日召回，需要有历史兴趣。以下为且的关系：
1. 有历史兴趣的回流用户
2. 在兴趣弹窗中无行为 或 跳过兴趣弹窗
3. 回流当天没有浏览内容
4. 回流次日截止push发送时间之前不活跃

### 实现逻辑

#### 用户筛选流程
1. **回流用户识别**: 从 `lofter.dwd_user_return_di` 获取回流用户
2. **历史兴趣筛选**: 通过 `lofter_db_dump.ods_db_blog_settings_nd` 筛选有历史兴趣设置的用户
3. **兴趣弹窗行为分析**: 检查 `lofter.dwd_beginner_guide_page_events_di` 中的行为记录
4. **回流当天行为检查**: 通过 `lofter.ods_mda_app_raw_di` 检查是否有浏览行为
5. **次日活跃度检查**: 检查12:00前是否有任何活动

#### 内容推荐逻辑
1. **基于历史兴趣匹配**: 将用户历史兴趣标签与内容标签进行匹配
2. **内容质量筛选**: 选择最近30天内hot_30d > 0的内容
3. **去重处理**: 排除已曝光、用户禁用标签、用户自己创作的内容
4. **推送控制**: 每用户推送1条最符合的内容，7天内去重

### 关键特性
- **精准定向**: 基于用户历史兴趣进行个性化推荐
- **行为筛选**: 确保推送给真正需要召回的用户
- **质量保证**: 多层内容质量和合规性筛选
- **防打扰**: 严格的历史推送去重和频控

## 策略二：无明确兴趣与行为的新回用户热搜内容推荐

### 基本信息
- **文件位置**: `analysis/jobs/push-1930/`
- **调度时间**: 19:30
- **MessageType**: 31
- **Priority**: 1
- **ExecutionId**: `new_return_user_hot_content`
- **Policy**: `new_return_user_hot_content`

### 策略描述
包含新回用户次日召回，无任何行为的也保证能被推送，理论上应该会覆盖所有未被PUSH触达到的新回用户。以下为且的关系：
1. 新回用户
2. 无历史兴趣
3. 新回当天没有任何有效行为
4. 新回次日截止push发送时间之前不活跃

### 实现逻辑

#### 用户筛选流程
1. **新回用户识别**: 合并 `lofter.dwd_user_new_di` 和 `lofter.dwd_user_return_di`
2. **无历史兴趣筛选**: 排除在 `lofter_db_dump.ods_db_blog_settings_nd` 中有兴趣设置的用户
3. **全面行为检查**: 检查浏览、互动、评论、搜索等所有有效行为
4. **兴趣引导检查**: 检查兴趣选择页面的所有操作
5. **次日活跃度检查**: 检查19:30前是否有任何活动

#### 内容推荐逻辑
1. **热搜内容获取**: 从 `lofter_dm.ads_hot_search_list_di` 获取热搜榜单内容
2. **榜单筛选**: 选择主要热搜榜单(listName 1-4)的前20名内容
3. **内容时效性**: 选择最近7天发布的内容
4. **质量保证**: 确保内容已发布、非违规、公开可见
5. **智能分配**: 按热搜排名和热度为用户分配最合适的内容

### 关键特性
- **全覆盖**: 覆盖所有无兴趣无行为的新回用户
- **热门内容**: 基于实时热搜榜单推荐当前最受关注的内容
- **兜底策略**: 作为其他推送策略的补充，确保用户不被遗漏
- **时效性强**: 推荐最新最热的内容，提升用户兴趣

## 技术实现要点

### 数据表结构
两个策略分别创建了对应的中间表：
- `lofter_dm.ads_push_return_user_history_interest_di`
- `lofter_dm.ads_push_new_return_user_hot_content_di`

### 时间参数使用
- `${azkaban.flow.1.days.ago}`: 回流/新用户当天
- `${azkaban.flow.current.date}`: 推送执行当天（次日）
- `${azkaban.flow.30.days.ago}`: 内容时间范围控制

### 性能优化
- 使用 `conf.spark.dynamicAllocation.maxExecutors=50` 提升并行度
- 合理的分区裁剪和Join优化
- 通过row_number()窗口函数控制推送数量

### 依赖关系
两个策略都采用双作业模式：
1. 数据准备作业 (`*_di.job`) 负责用户筛选和内容匹配
2. 推送作业 (`lofter_push_*.job`) 负责格式化和推送
