## 1. 服务概述

AI Service 是 AIGW（AI Gateway）的统一接入层，提供：

- **统一鉴权**：通过 API Key 代理访问 AIGW，业务侧无需持有底层 AIGW 凭证
- **用量追踪**：按 App、用户维度记录 token 消耗和请求明细
- **文件存储**：配套 NOS 上传能力，支持多模态场景
- **CORS 支持**：可从浏览器端直接调用（需提前配置允许的域名）

```
业务代码
   │  Authorization: Bearer <your-api-key>
   ▼
AI Service  (鉴权 + 用量记录 + CORS)
   │  app_id / app_key（内部持有）
   ▼
AIGW（大模型网关）
   │
   ▼
LLM（Claude / 其他模型）
```

## 2. 接入流程总览

| 步骤 | 操作方 | 说明 |
|------|--------|------|
| 1 | 申请者 → 管理员 | 提交接入申请（业务名称、使用场景） |
| 2 | 管理员 → AIGW 平台 | 在 AIGW 平台申请 App，获得 `app_id` / `app_key` |
| 3 | 管理员 → AI Service | 在本平台录入 AIGW App 凭证 |
| 4 | 管理员 → AI Service | 创建 API Key（personal 或 service 类型） |
| 5 | 管理员 → 申请者 | 通过安全渠道将 API Key 发送给申请者 |
| 6 | 申请者 | 将 API Key 配置到业务服务（环境变量 / K8s Secret） |
| 7 | 申请者 → AI Service | 调用 `/v1/*` 接口，验证接入正常 |

> **注意**：API Key 只在创建时完整展示一次，请妥善保管。

## 3. Step 1：确认需求，选择 Key 类型

在申请前，先明确你的使用场景，选择合适的 Key 类型：

### Personal Key（个人密钥，前缀 `sk-`）

| 适用场景 | 特点 |
|---------|------|
| 个人开发调试 | 绑定到具体用户邮箱 |
| 探索性实验 | 用量归因到个人 |
| 非生产环境测试 | 不建议用于生产服务 |

### Service Key（服务密钥，前缀 `svc-`）

| 适用场景 | 特点 |
|---------|------|
| 后端服务、API 调用 | 绑定到业务系统名称 |
| 生产环境部署 | 支持多用户共用 |
| 自动化流水线 | 需通过 `X-User-Code` 传递用户标识 |

> **推荐**：生产环境一律使用 Service Key，通过 `X-User-Code` 实现细粒度用量追踪。

## 4. Step 2：联系管理员申请 App 和 API Key

### 4.1 你需要提供的信息

向管理员提交以下申请材料：

**新业务接入（首次）：**

```
业务名称：<系统/模块名称>
使用场景：<简述用途，例如：用于内容推荐，调用 claude-3-5-haiku 模型>
Key 类型：service（推荐）或 personal
是否需要用户追踪：是（默认）/ 否
申请人：<姓名 + 邮箱>
```

> App Code 和 AIGW 凭证由管理员在 AIGW 平台申请后确定，申请时无需填写。

**已有 App，追加 Key：**

```
App Code：<已有的 appCode>
Key 类型：service 或 personal
服务名称：<业务系统名称，service key 必填>
申请人：<姓名 + 邮箱>
```

### 4.2 管理员操作说明

管理员收到申请后，分两步完成：

**第一步：在 AIGW 平台申请 App 凭证（如需新建 App）**

前往 AIGW 管理平台，按业务需求申请一个新的 App，获得平台分配的：
- `app_id`：应用 ID（ASCII 英文标识符，如 `lofter_tech`）
- `app_key`：应用密钥

> 如果业务可以复用已有 App 的凭证，跳过此步骤直接创建 Key。

**第二步：在本平台录入 AIGW App 凭证（如需新建 App）**

登录管理后台（`/admin`），进入「应用管理」，录入从 AIGW 平台获取的凭证：
- 填写 App Code（自定义英文标识，全局唯一，如 `lofter_feed`）
- 填入 AIGW 分配的 `app_id` 和 `app_key`
- 填写业务名称和描述
- 设置是否强制要求用户标识（`require_user_code`，默认开启）

> 录入时系统会自动验证 AIGW 凭证的有效性，凭证错误会立即提示。

**第三步：创建 API Key**
- 选择 Key 类型（`personal` / `service`）
- 选择归属的 App
- Service Key：填写业务系统名称
- Personal Key：填写目标用户邮箱（或由管理员代填 `target_name`）

### 4.3 你会收到的信息

管理员会将以下信息发送给你（请通过安全渠道传输）：

```
API Key：sk-{appCode}-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
         或
         svc-{appCode}-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

App Code：{appCode}
服务地址：https://{your-service-domain}
```

> ⚠️ **API Key 只展示一次**，遗失需要重新创建。请立即保存到密钥管理系统（如 Vault、K8s Secret）。

## 5. Step 3：发起 AI 请求

### 5.1 基础调用

AI Service 兼容 OpenAI API 格式，所有请求发送到 `/v1/*` 路径：

**请求格式：**

```
POST https://{service-domain}/v1/messages
Authorization: Bearer <your-api-key>
Content-Type: application/json
```

**示例：调用 Claude Messages API**

```bash
curl https://{service-domain}/v1/messages \
  -H "Authorization: Bearer sk-lofter-xxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "claude-3-5-haiku-20241022",
    "max_tokens": 1024,
    "messages": [
      {"role": "user", "content": "你好，请介绍一下你自己"}
    ]
  }'
```

**响应示例：**

```json
{
  "id": "msg_01XFDUDYJgAACzvnptvVoYEL",
  "type": "message",
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "你好！我是 Claude..."
    }
  ],
  "model": "claude-3-5-haiku-20241022",
  "stop_reason": "end_turn",
  "usage": {
    "input_tokens": 15,
    "output_tokens": 120
  }
}
```

### 5.2 Service Key：传递用户标识

使用 Service Key 且 App 配置了 `require_user_code=true` 时，**必须**在请求头中携带用户标识：

```bash
curl https://{service-domain}/v1/messages \
  -H "Authorization: Bearer svc-lofter-xxxxxxxxxxxx" \
  -H "X-User-Code: user@corp.com" \
  -H "Content-Type: application/json" \
  -d '{"model": "claude-3-5-haiku-20241022", "max_tokens": 512, "messages": [...]}'
```

`X-User-Code` 可以是：
- 用户邮箱（如 `alice@corp.com`）
- 用户 ID（如 `user_12345`）
- 其他业务系统中能唯一标识用户的字符串

> 缺少 `X-User-Code` 时，服务将返回 `400 Bad Request`。

### 5.3 流式响应（Streaming）

```bash
curl https://{service-domain}/v1/messages \
  -H "Authorization: Bearer <your-api-key>" \
  -H "Content-Type: application/json" \
  -H "X-User-Code: user@corp.com" \
  -d '{
    "model": "claude-3-5-haiku-20241022",
    "max_tokens": 1024,
    "stream": true,
    "messages": [
      {"role": "user", "content": "请写一篇关于 AI 的文章"}
    ]
  }'
```

响应为 `text/event-stream` 格式，服务会透传 AIGW 的流式输出。

### 5.4 代码示例

**Python（使用 anthropic SDK）：**

```python
import anthropic

client = anthropic.Anthropic(
    api_key="sk-lofter-xxxxxxxxxxxx",
    base_url="https://{service-domain}",
)

message = client.messages.create(
    model="claude-3-5-haiku-20241022",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello, Claude"}],
)
print(message.content)
```

**JavaScript/TypeScript（使用 anthropic SDK）：**

```typescript
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic({
  apiKey: "svc-lofter-xxxxxxxxxxxx",
  baseURL: "https://{service-domain}",
  defaultHeaders: {
    "X-User-Code": "user@corp.com",
  },
});

const message = await client.messages.create({
  model: "claude-3-5-haiku-20241022",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Hello, Claude" }],
});
console.log(message.content);
```

**Node.js（fetch）：**

```javascript
const response = await fetch("https://{service-domain}/v1/messages", {
  method: "POST",
  headers: {
    Authorization: "Bearer svc-lofter-xxxxxxxxxxxx",
    "Content-Type": "application/json",
    "X-User-Code": "user_12345",
  },
  body: JSON.stringify({
    model: "claude-3-5-haiku-20241022",
    max_tokens: 1024,
    messages: [{ role: "user", content: "你好" }],
  }),
});

const data = await response.json();
```

## 6. Step 4：文件上传（可选）

如需在 AI 请求中包含文件内容（图片、文档等），可通过 NOS 上传能力获取文件 URL，再传给 AIGW。

### 6.1 获取上传凭证

```bash
POST https://{service-domain}/api/nos/token
Authorization: Bearer <your-api-key>
Content-Type: application/json

{
  "file_name": "screenshot.png",
  "mime_type": "image/png"
}
```

**响应：**

```json
{
  "token": "NOS上传Token",
  "endpoint": "https://nos-eastchina1.126.net",
  "bucket": "your-bucket",
  "objectKey": "uploads/user@corp.com/2024/01/01/xxxxxxxx.png",
  "expires": 1704067200,
  "expiresAt": "2024-01-01T00:00:00.000Z",
  "uploadLogId": "uuid-xxxx-xxxx",
  "uploadUrl": "https://nos-eastchina1.126.net/your-bucket/...",
  "fileUrl": "https://nos-eastchina1.126.net/your-bucket/...",
  "params": { ... }
}
```

### 6.2 上传文件到 NOS

使用返回的 `uploadUrl` 和 `params` 直接 PUT 上传：

```bash
curl -X PUT "{uploadUrl}" \
  -H "Content-Type: image/png" \
  -H "Authorization: NOS {token}" \
  --data-binary @screenshot.png
```

### 6.3 通知上传完成

```bash
POST https://{service-domain}/api/nos/upload-complete
Authorization: Bearer <your-api-key>
Content-Type: application/json

{
  "upload_log_id": "uuid-xxxx-xxxx"
}
```

成功响应：

```json
{
  "success": true,
  "objectKey": "uploads/...",
  "bucket": "your-bucket"
}
```

### 6.4 在 AI 请求中使用文件 URL

上传完成后，将 `fileUrl` 用于多模态请求：

```json
{
  "model": "claude-3-5-sonnet-20241022",
  "max_tokens": 1024,
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image",
          "source": {
            "type": "url",
            "url": "https://nos-eastchina1.126.net/your-bucket/..."
          }
        },
        {
          "type": "text",
          "text": "请描述这张图片的内容"
        }
      ]
    }
  ]
}
```

## 7. 认证与安全须知

### 7.1 API Key 安全规范

| 规范 | 说明 |
|------|------|
| **不要硬编码** | Key 不能写入代码仓库、不能明文写在配置文件中 |
| **使用密钥管理** | 通过环境变量或 K8s Secret 注入 |
| **定期轮换** | 建议每 90 天更换一次 Service Key |
| **最小权限** | 不同业务使用不同的 Key，不共用 |
| **及时吊销** | 人员离职或 Key 泄露时立即联系管理员吊销 |

### 7.2 CORS（浏览器端调用）

如果需要从浏览器端（前端页面）直接调用 AI Service，需要管理员在环境配置中将你的域名加入 `CORS_ALLOWED_ORIGINS` 白名单。

申请时请提供：
- 前端页面域名（如 `https://lofter.com`、`https://staging.lofter.com`）

### 7.3 超时说明

| 场景 | 超时时间 |
|------|---------|
| 默认请求超时 | 120 秒 |
| 超时响应码 | `504 Gateway Timeout` |

建议客户端设置略大于 120 秒的超时（如 130 秒），避免竞态。

## 8. 错误码说明

| HTTP 状态码 | 含义 | 处理建议 |
|-------------|------|---------|
| `400 Bad Request` | 请求参数错误（如缺少 `X-User-Code`） | 检查请求头和参数 |
| `401 Unauthorized` | API Key 无效、缺失或已吊销 | 检查 `Authorization` 头，联系管理员 |
| `403 Forbidden` | 无权访问（越权操作） | 确认 Key 权限 |
| `404 Not Found` | 资源不存在 | 检查路径或资源 ID |
| `409 Conflict` | 资源冲突（如重复创建） | 检查是否已存在 |
| `502 Bad Gateway` | AIGW 网关异常 | 稍后重试，或联系管理员 |
| `504 Gateway Timeout` | AIGW 请求超时 | 稍后重试，检查请求复杂度 |

## 9. 常见问题

**Q：API Key 丢失怎么办？**  
A：联系管理员吊销旧 Key，重新创建新 Key。Key 只在创建时完整展示，无法找回。

**Q：Service Key 调用时返回 400，提示 `X-User-Code header is required`？**  
A：该 App 配置了强制用户追踪，必须在请求头中添加 `X-User-Code: <用户标识>`。

**Q：可以用 Personal Key 跑生产服务吗？**  
A：不建议。Personal Key 绑定个人用户，用量归因不清晰，建议生产环境统一使用 Service Key。

**Q：模型调用返回 502？**  
A：下游 AIGW 出现异常，请稍等片刻后重试。如持续出现，联系管理员排查。

**Q：如何查看我的 token 用量？**  
A：需要具有管理后台访问权限。你也可以联系管理员提供用量报告（按 App Code 或用户维度）。

**Q：能支持哪些模型？**  
A：AI Service 透传所有 AIGW 支持的模型，具体可用模型列表取决于你的 App 在 AIGW 侧的权限，请咨询管理员。
